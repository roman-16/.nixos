# Homelab N100 Server

## Overview

Add a second NixOS machine (Intel N100 server) to this repo, running isolated VMs for home services (Home Assistant OS, OpenClaw, and future services). Managed entirely from this PC via remote deployment over SSH.

## Hardware

- **CPU**: Intel N100 (4 cores, 4 threads, Alder Lake-N E-cores)
- **RAM**: 16GB recommended (minimum for running multiple VMs)
- **Power**: 6-9W idle with NixOS
- **Virtualization**: VT-x and VT-d supported (required for KVM/IOMMU)

## Architecture Decision

### Chosen approach: NixOS + microvm.nix + NixVirt

After researching all options (Proxmox, Docker, Incus, NixOS containers, microvm.nix), this combination was chosen because:

- **Fully declarative** -- everything defined in Nix, version-controlled in git
- **VM isolation** -- each service runs in its own VM with a separate kernel
- **HAOS support** -- NixVirt provides declarative libvirt VM definitions for non-NixOS guests
- **Docker support** -- Docker runs inside microvm.nix NixOS guests via `virtualisation.oci-containers`
- **Single repo** -- both this PC and the N100 managed from one flake

### Component responsibilities

| Component | Role |
|---|---|
| **NixOS** | Host OS on the N100 bare metal |
| **microvm.nix** | Declarative lightweight NixOS VMs (for Docker services, future NixOS services) |
| **NixVirt** | Declarative libvirt/QEMU VM definitions (for HAOS and any non-NixOS guest) |

### What runs where

| Service | Method | Why |
|---|---|---|
| Home Assistant OS | NixVirt (libvirt VM from `.qcow2`) | HAOS needs its own OS for Supervisor, add-ons, USB passthrough |
| OpenClaw | microvm.nix VM with Docker | NixOS guest running Docker container |
| Future NixOS services | microvm.nix VMs | Declarative, isolated, lightweight |
| Future non-NixOS services | NixVirt VMs | For anything that needs a specific OS image |

### RAM budget (16GB)

- NixOS host: ~1-2GB
- HAOS VM: 2GB
- microvm.nix VMs: 1-2GB each
- Remaining: ~8-10GB free for additional services

## Alternatives considered and rejected

| Option | Why rejected |
|---|---|
| **Proxmox** | Not declarative. GUI-first, state drift, can't version control VM definitions |
| **NixOS + Incus** | Incus VM/container management is imperative (CLI commands, not Nix config) |
| **microvm.nix only** | Guests must be NixOS -- can't run HAOS |
| **NixOS containers only** | Shared kernel (weaker isolation), can't run HAOS |
| **Separate repo** | Duplicates flake inputs, shared settings, and module code. No benefit |

### hosts/homelab/configuration.nix skeleton

```nix
{ inputs, lib, pkgs, ... }: {
  imports = [
    ./hardware-configuration.nix
    ./vms/haos.nix
    ./vms/openclaw.nix
  ];

  boot.loader = {
    systemd-boot.enable = true;
    efi.canTouchEfiVariables = true;
  };

  networking = {
    hostName = "homelab";
    useNetworkd = true;
  };

  # Bridge for VMs to access the LAN
  systemd.network = {
    enable = true;
    networks."10-lan" = {
      matchConfig.Name = [ "enp*" "vm-*" ];
      networkConfig.Bridge = "br0";
    };
    netdevs."br0" = {
      netdevConfig = {
        Kind = "bridge";
        Name = "br0";
      };
    };
    networks."10-br0" = {
      matchConfig.Name = "br0";
      networkConfig.DHCP = "yes";
    };
  };

  # libvirt for NixVirt (HAOS VM)
  virtualisation.libvirtd = {
    enable = true;
    qemu.package = pkgs.qemu_kvm;
  };

  nix = {
    optimise.automatic = true;
    nixPath = [ "nixpkgs=${inputs.nixpkgs}" ];
    settings.experimental-features = [ "nix-command" "flakes" ];
  };

  nixpkgs.config.allowUnfree = true;

  # Allow SSH for remote deployment
  services.openssh = {
    enable = true;
    settings.PermitRootLogin = "yes"; # needed for nixos-rebuild --target-host
  };

  users.users.roman = {
    isNormalUser = true;
    extraGroups = [ "wheel" "libvirtd" ];
    openssh.authorizedKeys.keys = [
      # TODO: add your SSH public key here
    ];
  };

  system.stateVersion = "24.05";
}
```

```nix
    autostart = true;
    domain = {
      type = "kvm";
      name = "haos";

        disk = [{
          type = "file";
```

### hosts/homelab/vms/openclaw.nix skeleton

```nix
# OpenClaw -- runs as Docker container inside a microvm.nix NixOS VM
{
  microvm.vms.openclaw = {
    config = { pkgs, ... }: {
      microvm = {
        vcpu = 2;
        mem = 2048;
        interfaces = [{
          type = "tap";
          id = "vm-openclaw";
          mac = "02:00:00:00:00:02";
        }];
        volumes = [{
          mountPoint = "/var/lib/openclaw";
          image = "openclaw.img";

      virtualisation.oci-containers.containers.openclaw = {
        image = "ghcr.io/steipete/openclaw:latest";
```

## Deployment

### Commands (run from this PC)

```bash
# Deploy to this PC (unchanged)
sudo nixos-rebuild switch --flake .#roman-nixos

# Deploy to the N100 over SSH
nixos-rebuild switch --flake .#homelab --target-host root@<n100-ip>

# Build without deploying (to check for errors)
nixos-rebuild build --flake .#homelab
```

### Initial N100 setup (one-time)

1. Boot NixOS installer on the N100 (USB stick)
2. Install minimal NixOS (just networking + SSH + user)
3. Copy the generated `hardware-configuration.nix` from the N100 to `hosts/homelab/`
4. Download the HAOS `.qcow2` image onto the N100 at `/var/lib/libvirt/images/haos.qcow2`
5. From this PC: `nixos-rebuild switch --flake .#homelab --target-host root@<n100-ip>`

### Updating

```bash
# Update flake inputs (shared lock file)
nix flake update

# Then deploy to whichever machines you want (independently)
sudo nixos-rebuild switch --flake .#roman-nixos       # this PC
nixos-rebuild switch --flake .#homelab --target-host root@<n100-ip>  # N100
```

Updating the lock file does NOT auto-deploy anywhere. Each machine is rebuilt explicitly.

## Known gotchas

- **microvm.nix first build**: Kernel compilation is slow if not cached. Use `cachix` or Nix binary cache
- **microvm.nix boot race condition**: ~1/10 boots a virtiofs mount can fail. Mitigate with systemd restart-on-failure and increased timeouts
- **HAOS disk image is stateful**: NixVirt declares the VM definition declaratively, but the `.qcow2` image itself is managed by HAOS internally (updates, config, add-ons)
- **NixVirt API**: Verify the exact NixVirt Nix API before implementing -- the structured domain definition may differ from the skeletons above. Check the repo README and examples
- **Networking**: Bridge setup ordering matters with systemd-networkd. The skeleton above uses priority prefixes ("10-lan", "10-br0") to control order
- **USB passthrough for HAOS**: Requires IOMMU enabled in BIOS and `intel_iommu=on` in kernel params. PCI/USB device IDs must match your hardware
